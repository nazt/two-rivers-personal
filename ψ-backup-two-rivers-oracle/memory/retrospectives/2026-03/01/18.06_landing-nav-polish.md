## AI Diary

This session was a study in iteration speed vs. design stability. The first half went smoothly — monitoring issue #10, capturing screenshots, updating stats, cleaning up CSS. Standard maintenance work. The second half was a rapid UI iteration loop that I'm not fully proud of.

The sticky header saga was particularly instructive. Nat wanted a sticky workshop stats header, but it caused scroll jank. We tried: remove sticky → add it back plain → make it sticky with solid background → morph to compact line → too complex, revert to simple. Five commits of back-and-forth. I should have asked more clarifying questions upfront instead of implementing each variation.

The dot navigation was a simpler version of the same pattern. The generic `nav {}` CSS selector was a trap — it styled ALL nav elements, and I didn't catch it until two deploys later. Eventually I just changed the dot-nav from `<nav>` to `<div>`. Sometimes the simplest fix is the right one.

The Report button scroll behavior worked well conceptually but the centering took three tries. `inline-flex` with `align-items: center` was the answer — I should have used it from the start instead of trying `text-align`, `line-height`, etc.

16 commits in this session. Too many for the amount of actual change. I need to prototype more locally before deploying.

## Honest Feedback

**1. Too many deploy cycles for CSS tweaks.** 16 commits where 5-6 would have sufficed. Each sticky header experiment was a separate commit+push+deploy cycle. I should batch related changes and test locally first, or at minimum combine "fix X" commits when they're addressing the same feature within minutes.

**2. Generic CSS selectors are landmines.** The `nav {}` selector affecting the dot-nav was predictable in hindsight. When I added a new `<nav>` element, I should have immediately checked what CSS rules target `nav` globally. This cost 2 extra deploy cycles and user frustration.

**3. Should ask "what does done look like?" earlier.** The sticky header went through 5 states because I kept implementing before fully understanding what Nat wanted. A quick mockup description or asking "sticky or just scrolls normally?" would have saved 4 commits.

## Lessons Learned

- Generic element selectors (`nav`, `a`, `div`) in CSS will bite you when you add new elements of the same type — always scope with classes
- When iterating on UX with fast deploy cycles, batch fixes before pushing — test locally first
- `IntersectionObserver` with `boundingClientRect.top < 0` distinguishes "scrolled past" from "hasn't reached yet"
