## Honest Feedback

**Friction 1: Playwright installation dance.** The batch screenshot approach requires Playwright browsers installed. We hit this twice — once trying to capture from the wrong project directory, then needing to install Chromium. For a tool that's used across multiple Oracle repos, Playwright should either be globally available or there should be a shared script. The current approach of `npx playwright screenshot` downloading fresh each time is fragile.

**Friction 2: Screenshot batch capture is slow.** 25 pages × ~6 seconds each = 2.5+ minutes minimum. The batch loop was still running when we pivoted to Vercel. For the visual gallery to be maintainable, we need either: (a) a faster capture method, (b) incremental captures (only screenshot changed pages), or (c) accept that screenshots are a periodic batch job, not real-time.

**Friction 3: Context switching cost.** The session jumped between 4 different repos (two-rivers-oracle, midnight-muse-oracle, landing-oracle, two-rivers-personal) and 3 different tasks (deployment, research, Vercel fix). Each switch required re-orienting. The handoff plan helped, but mid-session pivots (visual gallery → Vercel fix) leave work in an intermediate state. The batch screenshots are still running/incomplete.

## Lessons Learned

1. **Screenshot capture is the missing automation** — The Oracle ecosystem has deployment automation (wrangler.json pattern), gallery registration (markdown + Zod), but no screenshot pipeline. This is the gap to fill.
2. **Vercel static sites need outputDirectory** — When HTML lives in a subdirectory (docs/), add `vercel.json` with `"outputDirectory": "docs"`. Three lines, instant fix.
3. **Deep learn validates architecture quickly** — 5 parallel Haiku agents can map an entire codebase's patterns in ~5 minutes, cheaper than reading everything manually. The "no screenshot automation" finding emerged from all 5 agents independently.
