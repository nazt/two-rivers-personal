## Honest Feedback

**Friction 1: ค่ะ/ครับ memory categorization.** The rule was in memory, first section, line 4. But because it was under "TTS / Voice Rules", I applied it only to TTS, not typed text. This is a systemic issue — memory organization affects recall patterns. The fix (renaming to "applies to ALL output") is a bandaid; the real lesson is that critical identity rules should be repeated in context, not just stored.

**Friction 2: MQTT port trial-and-error.** Tried 8884 (protocol error), 8883 (protocol error), then 1883 worked. The screenshot showed port 8884 which is WebSocket, but mosquitto_pub speaks native MQTT. I should have recognized immediately that the web dashboard uses WSS (8884) while CLI tools need plain MQTT (1883). This cost 30 seconds but felt clumsy in front of students.

**Friction 3: Double index.html rewrite.** I rewrote index.html as a nice workshop portal, then 90 minutes later had to rewrite it again as a landing page and move the workshop to workshop.html. If the plan had included "this will eventually need a proper landing page", I could have started with workshop.html from the beginning. The rename cascaded through 20+ files — manageable with sed, but unnecessary churn.

## Lessons Learned

1. **Memory categorization shapes behavior** — A rule filed under "TTS" won't apply to text. Name memory sections by their SCOPE (all output), not their origin (TTS discussion).
2. **MQTT port conventions** — 1883 = plain MQTT, 8883 = MQTT+TLS, 8884 = WSS. Web dashboards use WSS; CLI tools use plain/TLS.
3. **Plan for the landing page first** — When deploying to a custom domain, start with the landing page, not the content pages. The domain's root path matters most.
